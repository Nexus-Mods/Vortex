The method currently passes `exeName` (which is `'ModInstallerIPC.exe'` from `BaseIPCConnection.findExecutable()`) directly to `super.getExecutablePaths()` and to the Vortex path join.

Replace the current method body with:

```typescript
protected getExecutablePaths(exeName: string): string[] {
  // On Linux, the executable has no .exe extension — it is a .NET 9
  // self-contained ELF binary that runs directly (not via mono).
  const platformExeName = process.platform === "linux"
    ? exeName.replace(/\.exe$/i, "")
    : exeName;
  const paths = super.getExecutablePaths(platformExeName);
  paths.push(
    path.join(
      getVortexPath("package_unpacked"),
      "node_modules",
      "@nexusmods/fomod-installer-ipc",
      "dist",
      platformExeName,
    ),
  );
  return paths;
}
```

This is a surgical platform-conditional change per D-04:
- On Windows: `exeName` is unchanged (`ModInstallerIPC.exe`), behavior identical to before
- On Linux: `.exe` is stripped, so the binary resolves to `ModInstallerIPC` (the ELF)
- Stripping `.exe` also prevents `RegularProcessLauncher` from hitting the mono fallback path (line 948 in the package), which would fail because `ModInstallerIPC` is a native .NET 9 binary, not a mono target
