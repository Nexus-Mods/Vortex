Pitfall 2: libloot.so Not on LD_LIBRARY_PATH at Runtime
- The linker flag `-l../loot_api/libloot` creates a build-time path
- At runtime, the system linker may not find liblibloot.so
- Fix: add `-Wl,-rpath,'$$ORIGIN/../loot_api'` to Linux linker flags in binding.gyp
  (requires patch-package) OR set LD_LIBRARY_PATH in the verify/start scripts

**Step A: Diagnose RPATH issue**

After plans 03-01 and 03-02 are complete, run locally:
```bash
npx @electron/rebuild -f -v 39.8.0 --only loot
```
Then check if loot loads:
```bash
node -e "require('loot')"
```

If it fails with `liblibloot.so: cannot open shared object file`:
- Run `ldd` on the built loot.node to confirm: `ldd $(find node_modules -name 'loot.node' -path '*/Release/*' | head -1)`
- The `.so` is in `loot_api/` but loot.node is in `build/Release/` — the linker can't find it without RPATH.

**Step B: Fix — set LD_LIBRARY_PATH in CI and verify script**

The cleanest fix that avoids patching binding.gyp is to set `LD_LIBRARY_PATH` to include the loot_api directory when loading the addon.

Option 1 (preferred — no patch-package needed):
Update the CI "Verify native addons load" step to set LD_LIBRARY_PATH:
```yaml
- name: Verify native addons load
  if: runner.os == 'Linux'
  run: |
    LOOT_API_DIR=$(node -e "const p=require('path');console.log(p.join(p.dirname(require.resolve('loot/package.json')),'loot_api'))")
    LD_LIBRARY_PATH="$LOOT_API_DIR:${LD_LIBRARY_PATH:-}" node scripts/verify-addons.cjs
```

Update `scripts/verify-addons.cjs` to handle the loot LD_LIBRARY_PATH issue in-script:
Before the addons loop, add:
```javascript
// Ensure liblibloot.so is discoverable at runtime
// The loot binding.gyp links with -l../loot_api/libloot but sets no RPATH
if (process.platform === "linux") {
  try {
    const lootPkg = require.resolve("loot/package.json");
    const lootApiDir = path.join(path.dirname(lootPkg), "loot_api");
    const currentLdPath = process.env.LD_LIBRARY_PATH || "";
    if (!currentLdPath.includes(lootApiDir)) {
      process.env.LD_LIBRARY_PATH = lootApiDir + (currentLdPath ? ":" + currentLdPath : "");
    }
  } catch (e) {
    // loot not installed — will fail in the load check below
  }
}
```

NOTE: Setting `LD_LIBRARY_PATH` in-process before `require()` MAY work (dlopen checks the env var at call time on most Linux loaders). If it does NOT work (some systems cache the value), fall back to the CI wrapper approach above.

**Decision rule for Option 1 vs Option 2:** If verify-addons.cjs exits non-zero after applying
Option 1 (LD_LIBRARY_PATH in-process + CI wrapper) AND `ldd` on loot.node confirms
`liblibloot.so => not found`, switch immediately to Option 2 (patch-package). Do not retry
Option 1 more than once.

Option 2 (fallback — if LD_LIBRARY_PATH in-process doesn't work):
Use `patch-package` to add RPATH to loot's binding.gyp. Add to the `conditions` array:
```json
["OS=='linux'", {
  "ldflags": ["-Wl,-rpath,'$$ORIGIN/../loot_api'"]
}]
```
This embeds the relative path directly in loot.node. If this approach is needed:
1. `pnpm add -D patch-package --workspace-root`
2. Edit the binding.gyp in node_modules
3. `npx patch-package loot`
4. Add `patchedDependencies` entry to pnpm-workspace.yaml
5. Re-run `npx @electron/rebuild -f -v 39.8.0 --only loot`
